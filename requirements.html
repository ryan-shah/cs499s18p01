2.1

Technical Requirements (in order of priortiy)

  1. PHP 7.0 Webpage to manage which scripts are executed and what time they are executed
  2. MYSQL Database of scripts, dependencies, and schedules for Spydr to keep track of
  3. Run shell-executable scripts in parallel on a Linux server
  4. Identify dependencies of scripts and ensure they are run first
  5. Ability to schedule how frequently this series of scripts is run
  6. Script memory and runtime statistics and tracking during execution
  7. Dynamic load distribution of scripts over server resources based on user defined specifications
  8. Add/Edit list of known scripts

2.2

Effort and size estimation

The effort required to make a website which has all of the functionality which our customer wants (i.e. something that can edit a database, able to call an external program) is not going to be very high after we learn more about the Laravel framework. However, there will be a learning curve to understand the intricacies of Laravel and to make the layout look respectable. All of us have some experience in web development, so we won't have to take any time to get acquainted with the foundations of web programming (such as HTML and PHP). If we had to estimate, the basic functionality will take no more than two to three person weeks to implement.
Another component to this project is the server-side program that executes the scripts. We plan on writing this in C/C++, as it has to be high-performance for running memory and CPU intensive programs with minimal overhead. This program will be a tad complex to keep track of all of the dependencies and tracking the memory usage of the scripts it executes. Additionally, integrating it with the website and all of the programs that our customer will have additional complexity. Once we have access to the customer's server infrastructure, this should take two to three person weeks to implement as well.
Another crucial component is the database. Before we create anything, we will have to consider design choices which minimize memory usage and redundancy. Again, all of us have experience designing and creating databases, so most of the time spent here will be on coming up with good design choices. We should have to implement around three tables to include scripts, schedules, and dependencies. This should take no more than one person week.
There are several goals that we could continue to pour time into and continually refine. Storing performance metrics will not be difficult to do initially; there are a few terminal commands that will give this information to us at any time, but getting information from them without system calls may present an issue. Additionally, creating an algorithm to optimize the future nightly runs will not be trivial. The frontend of the website could also always be refined, giving consideration to design choices that are less resource-intensive while producing the same output.
